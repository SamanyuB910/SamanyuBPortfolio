This is a draft. The argument is in place; the prose is not.

The systems I am interested in here are the ones where failure does not look like failure. An ML model that reprices perishable inventory never crashes when it is wrong. It returns a number, the number is applied to a shelf tag, and the cost shows up later as margin loss or waste — far from the model, attributed to nothing. There is no stack trace to find, which means the usual testing instincts, built around catching exceptions, do not apply.

My claim is that for systems like this, slice-based evaluation is the equivalent of unit testing. An aggregate metric can look fine while a specific category in a specific region is consistently mispriced. Evaluating per-slice, across category and region segments, is what makes those failures visible at all.

The second tool is behavioral invariance tests: properties the model must satisfy regardless of its accuracy. A markdown should be monotonic in time-to-expiry — an item closer to expiring should never get a smaller discount than an identical item with more shelf life. Violations are bugs even when the loss curve looks healthy.

On a production retail system I worked on, this combination surfaced real failure modes and ended up guiding which data we curated next. The eval framework became the data strategy.
